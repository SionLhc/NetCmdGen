## 二、关键项目详情

### 2.1 画图引擎：drawio vs AntV X6

#### drawio（推荐生产环境）
- 仓库：https://github.com/jgraph/drawio
- 协议：Apache 2.0（**可商用**）
- 优势：
  - 开箱即用的网络拓扑形状库（Cisco / 通用网络图标）
  - 支持嵌入 iframe 模式 (`embed.diagrams.net`)
  - 1 亿+用户验证，稳定性极佳
  - JSON 格式可读、可程序化生成/解析
- 劣势：
  - 体积较大、定制深度需要熟悉 mxGraph
  - 二次开发学习曲线陡

#### AntV X6（推荐深度定制）
- 仓库：https://x6.antv.antgroup.com/
- 协议：MIT
- 优势：
  - 国产、轻量、文档中文友好
  - Vue/React 集成简单
  - 自定义节点（设备图标）很灵活
- 劣势：
  - 没有内置的网络设备图标库，需要自己做
  - 复杂拓扑功能（自动布局/对齐）不如 drawio 完善

**建议**：
- MVP 阶段用 X6 快速搭建（学习成本低）
- 需要"画图体验对标 drawio"再考虑切换或嵌入 drawio

---

### 2.2 命令模板引擎：Jinja2 生态

#### Jinja2（核心库）
- 标准 Python 模板引擎
- 网络自动化领域**事实标准**

#### ConfPlate（轻量参考）
- 仓库：https://github.com/ktbyers/ConfPlate
- 作者 ktbyers 是 Netmiko 作者，业界权威
- 一个最小可用的模板配置生成脚本

#### The-Network-Config-Generator（结构参考）
- 仓库：https://github.com/careed23/The-Network-Config-Generator
- 提供 Cisco IOS / Juniper JunOS 两套 Jinja2 模板
- 用 **YAML 库存 + Pydantic 校验 + Jinja2 渲染**的清晰架构
- 项目结构可直接借鉴：
  ```
  templates/
    cisco_base.j2
    juniper_base.j2
  routers.yaml      # YAML 库存
  generate.py       # 主引擎
  ```
- ⚠️ 仅支持 Cisco/Juniper，**华为/H3C/锐捷等国产模板需要自己写**（这正是本项目核心价值）

---

### 2.3 后端网络自动化：Netmiko / NAPALM / Nornir

#### Netmiko
- 仓库：https://github.com/ktbyers/netmiko
- 协议：MIT
- 功能：SSH 连接多厂商设备（华为/H3C/思科/Juniper/锐捷等近百种）
- 用途：未来扩展"一键下发命令到真机"功能

#### NAPALM
- 仓库：https://github.com/napalm-automation/napalm
- 协议：Apache 2.0
- 功能：抽象多厂商设备 API，支持配置部署/对比/回滚
- 用途：高级特性"配置审计/对比"

#### Nornir
- 仓库：https://github.com/nornir-automation/nornir
- 协议：Apache 2.0
- 功能：并发批量编排（替代 Ansible 的纯 Python 方案）
- 用途：批量配置场景

---

### 2.4 拓扑→drawio 自动转换

#### N2G (Network to Graph)
- 项目：https://github.com/dmulyalin/N2G
- 配合 Nornir 自动从 LLDP/CDP 发现并生成 drawio 图
- **反向价值**：本项目的"画图→命令"是正向，N2G 的"配置→图"可作为辅助功能

#### draw_network_plot
- 用 Python 程序化生成 drawio XML
- 适合"模板化拓扑"快速生成

---

### 2.5 配置验证：Batfish

- 仓库：https://github.com/batfish/batfish
- 协议：Apache 2.0
- 功能：多厂商配置静态分析（OSPF 邻居正确性、ACL 冲突、路由可达性）
- 用途：作为本项目的**高级会员功能**——生成命令前先做"配置仿真验证"

### 2.6 awesome 资源

- **awesome-network-automation**: https://github.com/networktocode/awesome-network-automation
  - 一站式索引，所有网络自动化生态项目都在这

## 三、技术栈调整建议

### 3.1 原规划 vs 调整后

| 模块 | 原规划 | 调整后 | 原因 |
|---|---|---|---|
| 后端语言 | Go (Gin) | **Python (FastAPI)** | Netmiko/NAPALM/Nornir 都是 Python，复用生态最大化 |
| 画图引擎 | AntV X6 | **AntV X6 (MVP) → drawio (生产)** | 分阶段切换 |
| 模板引擎 | Go template | **Jinja2** | 业界标准，大量现成模板 |
| 配置校验 | 自研 | **Pydantic** | The-Network-Config-Generator 已验证可行 |
| 设备下发 | (未规划) | **Netmiko (高级特性)** | 拉开与 qsxwl 的差距 |

### 3.2 调整后的架构图

```
[浏览器]
   │  HTTPS
   ▼
[Nginx]──静态资源
   │
   ▼
[FastAPI 网关]──┬──[鉴权服务] ── Redis
                ├──[拓扑服务] ── MySQL (graph_json)
                ├──[命令引擎] ── Jinja2 + 模板库 (YAML)
                │                ↓
                │              [Pydantic 参数校验]
                ├──[执行器] ──── Netmiko (可选，高级特性)
                ├──[校验器] ──── Batfish (可选，高级特性)
                └──[订单/资源服务]
```

## 六、需要警惕的"坑"

### 6.1 协议合规
- drawio (Apache 2.0)：**可商用**，需保留 NOTICE 文件
- Netmiko/Jinja2 (MIT)：最宽松，可商用
- Batfish (Apache 2.0)：可商用
- ⚠️ 检查 The-Network-Config-Generator 的 LICENSE 再决定是否抄结构（多数 MIT 类项目允许）

### 6.2 技术风险
- 国产厂商（迈普/海康/烽火/迪普/浪潮/博达/瑞斯康达/神州数码）社区资料**极少**，模板编写主要靠官方手册 + 真机/模拟器验证
- Netmiko 对部分国产设备支持不完善，需 fallback 到 Paramiko 直连

### 6.3 性能风险
- Jinja2 渲染本身很快，瓶颈在大批量场景的并发
- drawio 嵌入大型拓扑（>100 节点）会卡，需要分页/虚拟化

## 八、最终建议

1. **架构调整**：后端从 Go 改 Python（FastAPI），最大化复用 Netmiko/NAPALM/Jinja2 生态
2. **模板复用**：直接借鉴 The-Network-Config-Generator 的 YAML+Pydantic+Jinja2 架构
3. **画图先 X6 后 drawio**：MVP 用 X6 快速出，后期视体验需求决定是否切换
4. **核心壁垒在国产厂商模板**：13 厂商模板是 qsxwl 的护城河，本项目必须自研、且要做得更好（中文注释 + AI 解析）
5. **避免重复造轮子**：参考 awesome-network-automation 列表，**有成熟方案的就用，别浪费时间**
