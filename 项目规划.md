## 一、项目定位

### 1.1 目标用户
- 网络运维工程师（中小企业 IT、集成商现场工程师）
- 网络初学者、考证学员（HCIA/HCIP/CCNA/CCNP）

### 1.2 核心价值
**"画图勾选 → 一键生成多厂商命令"** —— 把繁琐的命令行工作图形化，降低多厂商配置门槛。

### 1.3 与原站差异化思路（避免直接侵权 + 形成竞争力）
| 维度 | 原站 qsxwl | 本项目（建议） |

### 2.2 核心功能模块
1. **拓扑画图器**：拖拽设备（路由器/交换机/防火墙/PC/监控）+ 连线
2. **参数勾选面板**：基于端口/IP/MAC 划 VLAN、聚合、DHCP、生成树、VRRP+MSTP、OSPF…
3. **多厂商命令引擎**：13 厂商（华为/H3C/思科/锐捷/迈普/海康/TP-LINK/烽火/迪普/浪潮/博达/瑞斯康达/神州数码）

### 2.3 命令覆盖矩阵（用户视角的功能列表）

> 💡 **核心理念**：用户不需要知道"特性码"是什么，"基础/VLAN/路由/安全"这些术语是开发者视角。
> 用户想的是："我要配一台出口路由器，帮我搞定上网、DHCP、端口映射、限速"。
> 下面的功能树按**用户能理解的工作任务**组织。

#### 交换机
- **接入交换机**：VLAN划分 + 端口分配 + 管理IP + SSH远程管理 + 端口安全
- **汇聚交换机**：VLAN间路由 + Eth-Trunk聚合 + STP生成树 + 链路冗余
- **核心交换机**：OSPF路由 + VRRP网关冗余 + DHCP中继 + ACL访问控制 + SNMP监控

#### 路由器 🔴 当前缺失较多
| 功能 | 用户场景 | 当前状态 |
|---|---|---|
| **WAN口上网** | PPPoE拨号 / 静态IP / DHCP获取 | ✅ RouterWanForm |
| **局域网DHCP** | 给内网PC自动分配IP | ❌ 缺失 |
| **NAT/端口映射** | 把内网服务器映射到公网 | ❌ ServiceForm 太通用 |
| **ACL访问控制** | 禁止某网段上网/限时上网 | ⚠️ SecurityForm 偏交换机 |
| **流控/带宽管理** | 限制每IP带宽 / 应用优先 | ⚠️ QoSForm 偏核心交换机 |
| **静态路由** | 多网段互通 | ✅ RoutingForm |
| **OSPF动态路由** | 大型网络自动学习路由 | ⚠️ RoutingForm 参数复杂 |
| **远程管理** | SSH / SNMP / NTP | ✅ BasicForm |
| **DNS代理** | 内网DNS解析 | ❌ 缺失 |
| **IP-MAC绑定** | 防ARP欺骗 | ⚠️ SecurityForm |

#### 防火墙
- 接口区域（Trust/Untrust/DMZ）
- 域间策略（放行/拒绝规则）
- NAT策略（SNAT/DNAT）
- 安全策略（应用识别、IPS、AV）
- VPN（IPSec/SSL VPN）

| 开发者思维 ❌ | 用户思维 ✅ |

## 三、技术架构

### 3.1 技术选型
| 层 | 选型 | 理由 |
|---|---|---|
| 前端 | **Vue 3 + Vite + Element Plus** | 国内生态成熟，与原站风格契合 |
| 画图引擎 | **AntV X6** 或 **drawio-mxgraph** | X6 更轻量、可定制；drawio 体验最佳 |
| 状态 | Pinia | Vue 官方推荐 |
| 后端 | **Python FastAPI** | 生态丰富、AI 集成方便、开发效率高 |
| 命令引擎 | **Adapter 模式 + Python 函数式**（详见§五） | 多厂商可扩展，比模板更灵活 |
| DB | 无需数据库（文件存储） | 拓扑图、配置文件本地 JSON 存储，降低运维成本 |
| 文件存储 | 本地 JSON 文件 | 用户拓扑图、生成的配置文件 |
| 部署 | Docker + Nginx + 阿里云轻量 | 成本可控 |

## 五、命令生成引擎（最核心模块）

### 5.1 设计思路
**特性 (feature) × 厂商 (vendor) → 模板 (template)** 的二维矩阵，模板用占位符渲染。

### 5.2 模板示例（YAML，仅作格式示意）
```yaml
feature: vlan_by_port
vendors:
  huawei:
    template: |
      vlan {{ vlan_id }}
       description {{ desc }}
      interface {{ port }}
       port link-type access
       port default vlan {{ vlan_id }}
    annotations:
      - "vlan {{ vlan_id }}: 创建 VLAN"
      - "port link-type access: 设置端口为接入模式"
  h3c:
    template: |
      vlan {{ vlan_id }}
       description {{ desc }}
      interface {{ port }}
       port link-type access
       port access vlan {{ vlan_id }}
  cisco:
    template: |
      vlan {{ vlan_id }}
       name {{ desc }}
      interface {{ port }}
       switchport mode access
       switchport access vlan {{ vlan_id }}
```

### 5.3 引擎流程
```
拓扑图 JSON  ─┐
参数勾选数据 ─┼─► 参数归一化 ─► 选择厂商模板 ─► 渲染 ─► 拼接注释 ─► 输出
设备型号    ─┘
```

### 5.4 复杂特性策略
- VRRP+MSTP、OSPF 等多设备协同特性 → 在拓扑层面识别"逻辑组"，遍历每台设备调用对应模板
- 提供 **DSL 校验**：参数缺失/冲突预警（例如 VLAN ID 范围、端口名格式）

### 6.4 拓扑与命令引擎的集成方案

```
用户操作流程：
┌──────────┐    ┌──────────┐    ┌──────────┐    ┌──────────┐
│ 拖入设备  │ → │ 配置属性  │ → │ 选择厂商  │ → │ 生成命令  │
│ 连线拓扑  │    │ 勾选特性  │    │ 切换对比  │    │ 复制导出  │
└──────────┘    └──────────┘    └──────────┘    └──────────┘
      ↓              ↓              ↓              ↓
 拓扑 JSON    参数 JSON      vendor 参数     命令文本
```

**关键集成点：**
1. 拓扑画布保存时输出标准 JSON（设备类型、连线关系、IP 分配）
2. 右侧参数面板根据设备类型动态展示可配置的特性（交换机↦VLAN/STP，路由器↦OSPF/BGP）
3. 点击"生成命令"时，遍历拓扑中的每个设备，根据其类型和角色调用对应 Adapter
4. 底部输出面板按厂商 Tab 分页显示，支持一键复制、对比查看

### 8.2 核心问题：Adapter 断连

**现象**：Adapter 层大量 `generate_*` 方法返回占位符字符串，但对应 `vendors/` 目录下的底层生成器代码已经完整实现。

```
Adapter 层（适配器接口）       Vendors 层（底层生成器）
─────────────────────────     ────────────────────────
HuaweiAdapter._gen_routing()   vendors/huawei/routing.py  ← 8KB 代码已就绪
  → 返回 "# routing 模板待接入"    HuaweiRoutingGenerator.gen_ospf()
                                HuaweiRoutingGenerator.gen_static_route()
                                
HuaweiAdapter._gen_security()  vendors/huawei/security.py  ← 23KB 代码已就绪
  → 返回 "# security 模板待接入"   HuaweiSecurityGenerator.gen_acl()
                                HuaweiSecurityGenerator.gen_nat()
                                
RuijieAdapter._gen_routing()   vendors/ruijie.py  ← 29 个完整方法
  → 返回 "! routing 模板待接入"     generate_ospf(), generate_bgp(), generate_stp()...
```

**影响**：约 60KB 的生成器代码处于闲置状态，只需完成 Adapter 层的"接线"即可激活。

**优先级**：⭐ 最高 — 不是写新代码，是桥接已有代码。预计 16 个方法，1 周可完成。

## 十一、风险与合规

### 11.1 法律风险（重要！）
- **不得直接复制**原站文案、图片、视频、CSS 样式表。所有 Hero 标语、卡片配图必须重新设计/原创撰写
- ENSP 模拟器、CRT 等**不能直接打包分发**，应改为"提供官方下载指引"
- 软著最好登记，避免反被诉抄袭
- 备案：ICP 备案 + 公安备案；涉及视频教学需考虑《网络视听许可证》（个人可先用第三方平台嵌入）

### 11.2 命令准确性风险
- 多厂商命令差异极大，**必须有真机/模拟器验证流程**
- 引入"命令模板版本"概念，错命令可快速回滚
- 提供"用户报错反馈"入口，社区共建

### 11.3 项目策略
- 纯模仿难有竞争力，**务必走差异化**（AI / 协作 / 开源生态）
- 完全开源免费，靠社区贡献驱动命令模板增长

### RouterOS 流控完整方案（规划）

真正的 ROS 流控需要 4 步配合：
1. **Mangle 标记** — 在 prerouting/postrouting 标记连接/包
2. **Route Mark** — 路由标记分离上行/下行
3. **Queue Tree** — 基于标记的队列树（parent=global）
4. **简单队列** — 作为后备/简易场景

```ros
# Mangle: 标记下载连接
/ip firewall mangle add chain=prerouting in-interface=ether1 connection-mark=no-mark action=mark-connection new-connection-mark=download

# Mangle: 标记上传包
/ip firewall mangle add chain=postrouting out-interface=ether1 action=mark-packet new-packet-mark=upload

# Queue Tree: 基于标记限速
/queue tree add name=Download parent=global packet-mark=download max-limit=50M
/queue tree add name=Upload   parent=global packet-mark=upload   max-limit=20M
/queue tree add name=VOIP     parent=Download packet-mark=voip    priority=1 limit-at=10M max-limit=50M
```

| # | 问题 | 根因 | 影响范围 | 复现步骤 |
