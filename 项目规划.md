## 一、项目定位

### 1.1 目标用户
- 网络运维工程师（中小企业 IT、集成商现场工程师）
- 网络初学者、考证学员（HCIA/HCIP/CCNA/CCNP）
- 网络专业学生、培训机构

### 1.2 核心价值
**"画图勾选 → 一键生成多厂商命令"** —— 把繁琐的命令行工作图形化，降低多厂商配置门槛。

### 1.3 差异化策略

| 维度 | 原站 qsxwl | 本项目 |
|---|---|---|
| 形态 | Web SaaS | Web SaaS + 离线桌面版 |
| 拓扑画图 | 较基础 | AntV X6 专业拓扑引擎 |
| AI 能力 | 无 | LLM 辅助（规划中） |
| 开放度 | 闭源付费 | 完全开源 MIT |
| 厂商数 | 13 | 8（华为/华三/锐捷/迈普/RouterOS/Cisco/TP-LINK/Juniper）→ 目标 13+ |

### 1.4 设计原则

| 开发者思维 ❌ | 用户思维 ✅ |
|---|---|
| "请选择 feature: basic + wan + nat + dhcp" | "我要配一台出口路由器" |
| 7个Tab分开配置 | 一个完整表单从上到下 |
| 报错 "KeyError: 'dest_network'" | "请填写静态路由的目的网段" |
| 模板名 "campus-core-switch" | "园区网核心交换机开局" |

## 二、技术架构

| 层 | 选型 | 理由 |
|---|---|---|
| 前端 | **Vue 3 + Vite + Element Plus** | 国内生态成熟 |
| 画图引擎 | **AntV X6** | 轻量可定制 |
| 状态管理 | Pinia | Vue 官方推荐 |
| 后端 | **Python FastAPI** | 生态丰富、AI 集成方便 |
| 命令引擎 | **Adapter 模式 + Python 函数式** | 多厂商可扩展 |
| 存储 | 本地 JSON 文件 | 无需数据库，降低运维成本 |
| 部署 | Docker + Nginx | 成本可控 |

## 三、命令生成引擎（核心模块）

### 3.1 设计思路
**特性 (feature) × 厂商 (vendor) → 模板 (template)** 的二维矩阵。

### 3.2 引擎流程
```
拓扑图 JSON ─┐
参数勾选数据 ─┼─► 参数归一化 ─► 选择厂商模板 ─► 渲染 ─► 拼接注释 ─► 输出
设备型号    ─┘
```

## 十九、RouterOS 流量监控 (RosConsole) — 已完成

> 目标：多设备 SNMP 流量对比监控看板
> 创建日期：2026-06-04 | 完成日期：2026-06-07
> 状态：✅ 已交付

### 19.1 最终交付方案

经过多轮迭代，最终方案为**纯 SNMP 只读轮询 + 多设备多接口独立卡片对比**。

| 阶段 | 方案 | 问题 | 最终解决 |
|---|---|---|---|
| v1 | REST API (443) 直连 | 需启用 www-ssl，安全风险高 | 放弃 |
| v2 | 原生 API (8728) | api 服务默认关闭，需额外配置 | 放弃 |
| v3 | SNMP SSE 实时推送 | SSE 连接不稳定，浏览器后台冻结 | 改为轮询 |
| v4 ✅ | SNMP 3s 轮询 + 滚动窗口 + 独立卡片 | 稳定、安全、可多设备对比 | **当前方案** |

### 19.2 核心架构

```
浏览器 (Vue3)
  ├── 设备管理（添加/删除/持久化）
  ├── 多选设备下拉 → 每设备一行接口标签
  ├── 独立监控卡片（每接口一张，含速率 + ECharts 曲线）
  └── setInterval 3s → fetch /api/ros/traffic/snapshot

FastAPI
  └── /api/ros/traffic/snapshot
       └── SNMP GET (pysnmp, 1.5s timeout)
            ├── ifInOctets.{ifIndex} → 累计接收字节
            └── ifOutOctets.{ifIndex} → 累计发送字节
                 └── 差分 → Mbps（后端内存缓存上一帧）
```

### 19.3 技术实现细节

| 特性 | 实现 |
|---|---|
| **采集频率** | 3 秒一次（避免 SNMP 超时压垮设备） |
| **显示窗口** | 最近 100 个数据点 ≈ 5 分钟趋势 |
| **后端 SNMP** | pysnmp 7.x async，每批 5 并发 + 间歇 0.3s |
| **前端轮询** | 递归 setTimeout（避开浏览器后台冻结） |
| **后台恢复** | `visibilitychange` 监听，切回可视标签页立马补采 |
| **超时处理** | fetch 5s AbortController + 接口列表 12s |
| **错误提示** | 连续失败 >2 次 → 卡片显示 `⚠ N 次失败` + 速度变灰 `—` |
| **多设备对比** | el-select multiple → 每设备独占一行接口标签 |
| **图表** | 每接口独立 echarts 实例，time 轴 + 蓝色下载/青色上传双线 |

> 问题：RosConsole 多设备多接口流量卡片图表，运行一段时间后停止更新
> 对比方案：Prometheus + snmp_exporter 的 Pull 模型架构

### 21.1 Prometheus SNMP 架构（不会停的监控）

```
┌──────────────────────────────────────────────────────────────┐
│                     Prometheus 架构                           │
│                                                              │
│  Grafana (前端)                                              │
│     ↑ 只读，即使 Grafana 挂了也不影响数据采集                    │
│  Prometheus Server                                           │
│     ↑ scrape (Pull) 每 30s/60s 拉一次                         │
│  snmp_exporter (独立进程/容器)                                 │
│     ↑ SNMP GET (独立轮询，不受前端影响)                         │
│  网络设备 (交换机/路由器)                                      │
│                                                              │
│  关键设计：采集层和后端解耦，采集进程独立运行永不停止              │
└──────────────────────────────────────────────────────────────┘

我们当前的架构（问题所在）：
┌──────────────────────────────────────────────────────────────┐
│  Vue3 浏览器 Tab                                             │
│     ↑ setInterval 3s → fetch /api/ros/traffic/snapshot       │
│  FastAPI 后端                                                │
│     ↑ 每请求一次 SNMP GET (pysnmp)                            │
│  网络设备                                                    │
│                                                              │
│  问题：浏览器 Tab 被后台 → JS 主线程挂起 → setInterval 停      │
│       前端的"心跳"断了，后端的"手臂"也就停了                    │
└──────────────────────────────────────────────────────────────┘
```

### 21.2 根因分析 — 四个层面的问题

| 层面 | 问题 | 症状 | 修复难度 |
|---|---|---|---|
| **浏览器层** | `setInterval` 在后台 Tab 被浏览器限频（1s→60s 甚至停止）| 切回来数据断层一段 | ⭐ 已修复 `visibilitychange` |
| **内存层** | ECharts 实例不断 push 数据点（3s×100点），长时间累积 → 内存泄漏 | 图表越来越卡 → 最终冻结 | ⭐⭐ |
| **后端层** | 每次请求新建 pysnmp session，无连接池/重用 | 高并发时 SNMP 超时累积 | ⭐⭐⭐ |
| **设备层** | RouterOS SNMP agent 在高频轮询下可能限流或超时 | 偶发丢包 → 数据断层 | ⭐⭐ |

### 21.3 借鉴 Prometheus 的修复方案

| 修复 | 对标 Prometheus | 具体做法 |
|---|---|---|
| **后端自治轮询** | snmp_exporter 独立运行 | FastAPI 后台启动 asyncio Task，定时采集 + 内存缓存，前端只读缓存 |
| **前端解耦** | Grafana 是纯读客户端 | 前端不再驱动采集节奏，只 fetch 最新缓存数据（被动读） |
| **数据滚动窗口** | Prometheus TSDB 自动压缩 | 后端内存保留最近 5 分钟数据（100 个点），超时自动丢弃 |
| **连接池复用** | exporter 长期持有 SNMP session | pysnmp 用 `asyncio` 的 `SNMPEngine` 长连接复用 |
| **并发控制** | scrape 间隔控制 | 后端每设备每 3s 一次，不同设备错开时间（stagger），避免并发峰值 |

### 21.4 推荐修复方案（最小改动，最大效果）

```
方案：后端自治轮询 + 前端被动读取

1. 后端新增 /api/ros/traffic/poll 启动后台采集任务
   - 使用 asyncio.create_task 在 FastAPI lifespan 中启动
   - 每 3s 对所有已注册设备 SNMP GET
   - 结果写入内存 dict: { device_id: { iface_name: [data_points] } }
   - 每个接口保留最近 100 个数据点（5 分钟窗口）

2. 前端改为被动读取
   - 仍然 setInterval 3s fetch /api/ros/traffic/snapshot
   - 但后端不再"即时采集"，只返回内存缓存的最新数据
   - 即使前端 Tab 被后台冻结 30 秒，回来后拉到的仍是连续数据

3. ECharts 数据窗口限制
   - setOption 时传递 notMerge: false + 限制 series.data.length ≤ 100
   - 超 100 点时 shift() 移除最早的点

4. pysnmp 连接池
   - 使用 asyncio.Queue 控制并发 (max 5)
   - 失败重试 1 次，不阻塞后续采集
```

### 24.4 已知问题（2026-06-07 晚，待明天排查）

| 问题 | 可能原因 |
|---|---|
| 部分页面功能无法使用 | 后端未启动/端口冲突/依赖缺失/前端热更新漏路由 |
| 侧边栏过長（22项） | 15 项运维工具可折叠二级菜单 |
| RosConsole 需重启 | 采集器依赖 `main.py` 的 `lifespan` 启动，未重启不生效 |
| Vite 热更新可能漏新路由 | 长时间 dev server 运行后新路由不加载 |
| batch 命令执行是模拟 | 需对接 SSH API 实现真实命令下发 |
| 态势大屏数据是模拟 | `/bigscreen/overview` 返回随机数据，需接真实 SNMP |
| 巡检/安全/防护结果是模拟 | 均需对接 SSH/SNMP 实现真实检查 |
| 前端刷新后页面白屏 | 确认后端 8000 端口已启动，前端 3000 端口已启动 |
