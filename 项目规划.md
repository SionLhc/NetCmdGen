## 一、项目定位

### 1.1 目标用户
- 网络运维工程师（中小企业 IT、集成商现场工程师）
- 网络初学者、考证学员（HCIA/HCIP/CCNA/CCNP）
- 网络专业学生、培训机构

### 1.2 核心价值
**"画图勾选 → 一键生成多厂商命令"** —— 把繁琐的命令行工作图形化，降低多厂商配置门槛。

### 1.3 差异化策略

| 维度 | 原站 qsxwl | 本项目 |
|---|---|---|
| 形态 | Web SaaS | Web SaaS + 离线桌面版 |
| 拓扑画图 | 较基础 | AntV X6 专业拓扑引擎 |
| AI 能力 | 无 | LLM 辅助（规划中） |
| 开放度 | 闭源付费 | 完全开源 MIT |
| 厂商数 | 13 | 8（华为/华三/锐捷/迈普/RouterOS/Cisco/TP-LINK/Juniper）→ 目标 13+ |

### 1.4 设计原则

| 开发者思维 ❌ | 用户思维 ✅ |
|---|---|
| "请选择 feature: basic + wan + nat + dhcp" | "我要配一台出口路由器" |
| 7个Tab分开配置 | 一个完整表单从上到下 |
| 报错 "KeyError: 'dest_network'" | "请填写静态路由的目的网段" |
| 模板名 "campus-core-switch" | "园区网核心交换机开局" |

## 二、技术架构

| 层 | 选型 | 理由 |
|---|---|---|
| 前端 | **Vue 3 + Vite + Element Plus** | 国内生态成熟 |
| 画图引擎 | **AntV X6** | 轻量可定制 |
| 状态管理 | Pinia | Vue 官方推荐 |
| 后端 | **Python FastAPI** | 生态丰富、AI 集成方便 |
| 命令引擎 | **Adapter 模式 + Python 函数式** | 多厂商可扩展 |
| 存储 | 本地 JSON 文件 | 无需数据库，降低运维成本 |
| 部署 | Docker + Nginx | 成本可控 |

## 三、命令生成引擎（核心模块）

### 3.1 设计思路
**特性 (feature) × 厂商 (vendor) → 模板 (template)** 的二维矩阵。

### 3.2 引擎流程
```
拓扑图 JSON ─┐
参数勾选数据 ─┼─► 参数归一化 ─► 选择厂商模板 ─► 渲染 ─► 拼接注释 ─► 输出
设备型号    ─┘
```

- 软著登记 + ICP 备案 + 公安备案

### 8.2 命令准确性
- 多厂商命令差异大，需真机/模拟器验证
- 引入命令模板版本概念
- 提供用户报错反馈入口

### 8.3 安全
- 端口扫描器需加白名单/限频
- Python 依赖建议锁定版本

## 十七、网络诊断系统升级方案（2026-06-04）

> 将现有散落在 Tools.vue 和 Diagnostics.vue 的诊断功能系统化升级为专业级诊断平台。

### 17.1 设计目标

在现有 Diagnostics.vue（PingProbe + TraceMap SSE 流式）基础上扩展更多诊断功能，每个功能独立子路由，所有功能支持 SSE 实时流式。

### 17.2 页面路由架构

```
/diagnostics                    → 诊断首页（功能入口卡片，显示最近状态）
/diagnostics/ping               → Ping 实时诊断（复用 PingProbe.vue）
/diagnostics/traceroute         → Traceroute 逐跳诊断（复用 TraceMap.vue）
/diagnostics/dns                → DNS 多解析器诊断（新增）
/diagnostics/tcp-port           → TCP 端口连通性检测（新增）
/diagnostics/http               → HTTP/HTTPS 可用性监测（新增）
/diagnostics/mtu                → MTU 路径发现（新增）
/diagnostics/jitter             → 网络抖动分析（新增）
/diagnostics/history            → 历史趋势对比（新增）
```
